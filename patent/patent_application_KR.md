
# 특허출원서

## [발명의 명칭]

**페이지 복잡도 사전 분류 기반 적응적 AI 처리 및 구조선 기반 읽기 영역 분할을 이용한 PDF 문서의 HTML/마크다운 변환·동시 번역 방법 및 시스템**

Method and System for Converting and Simultaneously Translating PDF Documents to HTML/Markdown Using Page Complexity Pre-Classification Based Adaptive AI Processing and Structural-Line-Based Reading Zone Segmentation

---

## [기술분야]

본 발명은 스캔된 이미지 PDF 문서 및 디지털 PDF 문서를 기계가 의미적으로 이해할 수 있는 구조화된 문서(HTML, 마크다운)로 변환하는 방법 및 시스템에 관한 것이다. 보다 상세하게는, (1) 각 페이지의 벡터 객체(선, 사각형) 및 텍스트 블록 분포를 로컬에서 고속 분석하여 페이지 복잡도를 0(텍스트 전용) 또는 1(복합)로 사전 분류하고, (2) 복잡도 태그에 따라 로컬 OCR 선처리와 다중모달 비전 AI를 적응적으로 분배하며, (3) 문서 내에 인간이 의도적으로 그어놓은 구조선(외곽선, 구분선, 표 테두리)을 기준으로 읽기 영역을 분할하는 적응적 하이브리드 문서 변환 방법에 관한 것이다. 나아가, 문서 변환과 동시에 다국어 번역을 단일 AI 호출 내에서 수행하여 추가 비용 없이 번역된 구조화 문서를 생성하는 방법에 관한 것이다.

## [배경기술]

### 종래 기술의 현황

PDF(Portable Document Format) 문서는 본질적으로 "이 글자를 이 좌표에 그려라"라는 렌더링 명령어의 집합이다. "이것이 제목이다", "이것이 표의 셀이다"와 같은 의미적 구조 정보가 내재되어 있지 않다. 이로 인해 PDF 문서를 HTML이나 마크다운과 같은 구조화된 문서로 변환하는 과정에서 다음과 같은 근본적인 기술적 난제가 존재한다.

### 종래 기술의 문제점

#### 문제점 1: 읽기 순서(Reading Order) 결정의 난제

종래 기술에서는 텍스트 블록의 XY 좌표를 기반으로 읽기 순서를 결정한다. 구체적으로, 텍스트 블록의 중심점 X좌표를 클러스터링하여 컬럼(단)을 감지하고, 각 컬럼 내에서 Y좌표 순으로 정렬하는 방식이다.

그러나 이 방식은 다음과 같은 근본적 한계를 갖는다:

(1) 다단 레이아웃에서 같은 Y좌표에 있는 텍스트가 좌-우 연속인지 별개의 컬럼인지 판별 불가
(2) 사이드바, 텍스트 상자, 각주가 혼합된 페이지에서 읽기 순서 오류 빈발
(3) 표(테이블) 내부의 셀과 외부의 텍스트 구분 불가
(4) 각주 영역과 본문 영역의 경계를 좌표만으로 판별하기 어려움

#### 문제점 2: 디지털 PDF 표(테이블) 텍스트 추출 시 내용 뒤섞임

디지털 PDF에서 텍스트 추출 라이브러리를 사용하여 텍스트를 직접 추출하면, 표의 외곽선(셀 구분선)은 버려지고 텍스트만 추출된다. 인간은 표의 각 셀 안에 문장을 넣어서 독자가 하나의 셀을 다 읽고 다음 셀(오른쪽 또는 아래쪽)로 이동하도록 외곽선으로 읽기 순서를 지시하는데, 그 외곽선이 제거되면 서로 다른 셀의 문장들이 뒤죽박죽으로 섞여버린다.

종래 기술에서는 이를 해결하기 위해 텍스트 좌표 간격으로 셀 경계를 추론하거나, 행/열 정렬 패턴을 분석하려 시도하지만, 셀 병합(rowspan/colspan), 선이 없는 표, 탭 들여쓰기와의 구분 등에서 심각한 오류가 발생한다.

#### 문제점 3: OCR에서의 한자 천간(天干) 문자 오인식

수능 시험, 변호사 시험, 공무원 시험, 계약서, 판결문 등 한국어 문서에서 당사자나 등장인물을 지칭하기 위해 갑(甲), 을(乙), 병(丙), 정(丁), 무(戊) 등의 한자 천간 문자가 빈번하게 사용된다. OCR 엔진은 이 문자들을 다음과 같이 오인식하는 경향이 있다:

- 乙(을) → Z, 2, 己
- 丁(정) → T, 7
- 甲(갑) → 田, 由
- 丙(병) → 内

이러한 오인식은 문서의 의미를 완전히 훼손한다.

#### 문제점 4: 비정형 지시선(Annotation Line)의 처리

문서에는 표의 테두리와 같은 구조적 선(수평/수직) 외에, 말풍선에서 본문으로의 화살표, 부연설명을 위한 지시선(대각선, 곡선) 등이 존재한다. 종래 기술에서는 이러한 선을 모두 동일하게 취급하거나 무시하여, 주석과 본문의 관계를 파악하지 못한다.

#### 문제점 5: 대규모 문서의 AI 처리 비용 및 속도 비효율

종래의 다중모달 비전 AI 기반 문서 변환 방법은, 모든 페이지를 일률적으로 동일한 방식(이미지 + 프롬프트를 AI에 전송)으로 처리한다. 그러나 일반적인 문서에서 텍스트만으로 구성된 단순 페이지는 전체의 60~80%를 차지하고, 표·그래프·이미지가 포함된 복잡한 페이지는 20~40%에 불과하다. 모든 페이지에 대해 고비용의 이미지 토큰을 AI에 전송하는 것은, 단순 텍스트 페이지에 대해서는 불필요한 비용과 지연을 발생시킨다.

구체적으로, 다중모달 비전 AI(예: Gemini, GPT-4V 등)에 페이지 이미지를 전송할 경우 페이지당 약 1,500개의 이미지 입력 토큰이 소비되는 반면, 동일한 페이지의 OCR 텍스트를 텍스트로만 전송할 경우 약 200개의 텍스트 토큰만 소비된다. 즉, 텍스트 전용 페이지에 이미지를 전송하는 것은 약 7.5배의 비용 낭비를 초래한다.

또한, 종래 기술에서는 하나의 문서를 변환하기 위해 읽기 순서 보정, 제목 분류, 텍스트 교정 등을 각각 별도의 AI 호출로 수행하여, 페이지당 3회 이상의 API 호출이 필요하였다. 이는 2,000페이지 규모의 대형 문서에서 600회 이상의 API 호출을 발생시켜, 처리 시간과 비용이 선형적으로 증가하는 문제가 있다.

## [발명의 내용]

### [해결하고자 하는 과제]

본 발명은 상기 종래 기술의 문제점을 해결하기 위한 것으로서, 다음의 기술적 과제를 해결하고자 한다.

첫째, 문서 내에 인간이 의도적으로 그어놓은 구조선(외곽선, 구분선, 표 테두리)을 읽기 영역 분할의 기준으로 사용하여, XY 좌표 기반 방식의 근본적 한계를 해결하는 것이다.

둘째, 디지털 PDF 문서에서 표(테이블)나 박스 등 구조선으로 둘러싸인 영역은 이미지 기반 AI 처리를 적용하고, 나머지 텍스트 영역은 디지털 텍스트 추출을 사용하는 하이브리드 처리 방법을 제공하는 것이다.

셋째, 벡터 라인의 기하학적 특성(각도)을 기반으로 구조선과 지시선(주석선)을 자동 분류하고, 지시선의 양 끝점을 추적하여 주석과 본문의 연결관계를 자동으로 파악하는 것이다.

넷째, 문서의 유형 및 맥락을 기반으로 한자 천간 문자의 OCR 오인식을 자동 보정하는 것이다.

다섯째, 각 페이지의 벡터 객체 및 텍스트 블록 메타데이터를 로컬에서 고속 분석하여 페이지를 복잡도 0(텍스트 전용) 또는 1(복합)로 사전 분류하고, 텍스트 전용 페이지에는 로컬 OCR 선처리 후 텍스트만을 AI에 전송하고, 복합 페이지에만 이미지를 AI에 전송하여, AI 처리 비용을 대폭 절감하면서도 변환 정확도를 유지하는 것이다.

여섯째, 읽기 순서 보정, 제목 분류, 텍스트 교정 등 복수의 분석 작업을 단일 AI 호출에 통합하여 수행하되, 페이지 복잡도 태그(0/1)와 스타일 메타데이터(폰트 크기, 볼드, 정렬)를 AI에 함께 전달하여 단일 호출만으로도 고정확도의 다중 작업 결과를 얻는 것이다.

일곱째, 문서 변환과 동시에 다국어 번역을 수행하되, 번역 지시를 기존의 변환용 AI 프롬프트에 통합하여 추가 API 호출 없이 단일 호출 내에서 변환과 번역을 동시에 수행하는 것이다. 이에 의해, 별도의 번역 API를 호출하는 종래 방식 대비 비용과 지연 시간이 대폭 절감된다.

### [과제의 해결 수단]

상기 과제를 해결하기 위한 본 발명의 방법은 다음의 단계를 포함한다.

#### 제1 발명: 구조선 기반 읽기 영역 분할 방법

본 발명에 따른 문서의 읽기 순서 결정 방법은:

(a) PDF 문서의 각 페이지에서 벡터 라인(vector line) 및 사각형(rectangle) 정보를 추출하는 단계;

(b) 추출된 벡터 라인을 기하학적 특성에 따라 구조선(structural line)과 지시선(annotation line)으로 분류하는 단계로서, 수평 방향(0도 +-5도) 또는 수직 방향(90도 +-5도)의 선을 구조선으로, 그 외 각도의 선을 지시선으로 분류하는 단계;

(c) 구조선과 사각형을 이용하여 페이지를 복수의 읽기 영역(zone)으로 분할하는 단계로서:
  - 사각형(rect)은 그 자체로 하나의 읽기 영역이 되고,
  - 수평 구분선은 페이지를 수평 밴드로 분할하고,
  - 수직 구분선은 밴드를 컬럼으로 분할하여,
  - 비중첩 격자(grid) 형태의 읽기 영역을 생성하는 단계;

(d) 각 텍스트 블록을 해당 블록의 바운딩 박스와 가장 높은 중첩 비율을 갖는 읽기 영역에 할당하는 단계;

(e) 읽기 영역을 콘텐츠 영역과 각주 영역으로 분류하되, 페이지 하단 75% 이하에 위치한 영역을 각주 영역으로 분류하는 단계;

(f) 콘텐츠 영역 내의 블록을 위에서 아래로, 왼쪽에서 오른쪽으로 정렬하고, 각주 영역의 블록을 콘텐츠 영역 이후에 배치하는 단계;

를 포함하며, 구조선이 감지되지 않는 페이지에 대해서는 텍스트 블록의 X좌표 클러스터링 기반 컬럼 감지 방식으로 폴백(fallback)하는 것을 특징으로 한다.

#### 제2 발명: 디지털 PDF 하이브리드 처리 방법

본 발명에 따른 디지털 PDF 문서의 변환 방법은:

(a) 디지털 PDF 문서의 각 페이지에서 디지털 텍스트 블록과 벡터 라인 정보를 동시에 추출하는 단계;

(b) 벡터 라인 중 구조선으로 분류된 선을 이용하여, 구조선으로 둘러싸인 영역(표, 박스 영역)을 탐지하는 단계;

(c) 구조선 영역이 탐지된 경우, 해당 페이지를 다음과 같이 하이브리드 방식으로 처리하는 단계:
  - (c-1) 디지털 텍스트 블록 중 구조선 영역과 50% 이상 중첩되는 블록을 제외하는 단계;
  - (c-2) 해당 페이지의 이미지를 이미지 기반 AI 파이프라인(레이아웃 감지 -> OCR -> 테이블 구조 인식)으로 처리하는 단계;
  - (c-3) AI 파이프라인 결과 중 구조선 영역과 30% 이상 중첩되는 블록만 선별하는 단계;
  - (c-4) 선별된 AI 처리 블록과 제외되지 않은 디지털 텍스트 블록을 병합하는 단계;

(d) 구조선 영역이 탐지되지 않은 경우, 종래의 디지털 텍스트 추출 방식으로 처리하는 단계;

를 포함하는 것을 특징으로 한다.

이에 의해, 디지털 PDF 문서에서 표 내부의 텍스트가 셀 구분 없이 뒤섞이는 종래 기술의 근본적 문제를 해결한다. 핵심 원리는, 인간이 표의 외곽선으로 독자에게 읽기 순서를 지시하는 것과 동일하게, AI도 외곽선을 먼저 인식한 후 그 내부의 텍스트를 처리하도록 하는 것이다.

#### 제3 발명: 벡터 라인 분류 및 지시선 기반 주석-본문 연결 방법

본 발명에 따른 벡터 라인 분류 방법은:

(a) PDF 페이지에서 추출된 각 벡터 라인에 대해, 시작점과 끝점의 좌표로부터 수평축 대비 각도(theta)를 계산하는 단계;

(b) theta가 5도 이하이거나 85도 이상인 경우 구조선(structural)으로, 그 외의 경우 지시선(annotation)으로 분류하는 단계;

(c) 지시선으로 분류된 라인에 대해, 한쪽 끝점이 주석 블록(말풍선, 화살표 주석, 각주) 근방(margin 30px 이내)에 위치하는지 판별하는 단계;

(d) 주석 블록 근방의 끝점이 발견된 경우, 반대쪽 끝점에서 가장 가까운 본문 블록(거리 80px 이내)을 탐색하여, 주석 블록과 본문 블록의 연결관계(linked_block_ids)를 자동으로 설정하는 단계;

를 포함하는 것을 특징으로 한다.

이 방법에 의해, 문서에서 대각선 화살표나 지시선으로 표현된 부연설명, 말풍선 주석, 첨삭 표시 등이 어떤 본문 내용을 가리키는지 자동으로 파악할 수 있다.

#### 제4 발명: 문맥 기반 한자 천간 문자 OCR 보정 방법

본 발명에 따른 OCR 보정 방법은:

(a) OCR 결과 텍스트에서 한자 천간 문자(甲, 乙, 丙, 丁, 戊, 己, 庚, 辛, 壬, 癸)와 혼동 가능한 문자(Z, T, 2, 7, 田, 由, 内 등)의 출현을 감지하는 단계;

(b) 문서의 유형을 판별하는 단계로서, 다음 중 하나 이상의 특징이 감지되면 "천간 빈출 문서"로 판별하는 단계:
  - 시험 문제지 관련 키워드 (문항, 정답, 배점, 수험번호 등)
  - 법률 문서 관련 키워드 (계약, 원고, 피고, 판결, 조항 등)
  - 관공서 양식 관련 키워드 (신청인, 피신청인, 등기 등)
  - 동일 문서 내 하나 이상의 정상 인식된 천간 문자의 존재;

(c) "천간 빈출 문서"로 판별된 경우, 문맥상 당사자/등장인물 지칭 위치에 출현하는 혼동 문자를 해당 천간 문자로 치환하는 단계로서:
  - "Z"을 "乙(을)"로 치환하는 경우
  - "T"를 "丁(정)"으로 치환하는 경우
  - 주어 위치("~은/는/이/가" 앞)에 출현하는 단독 알파벳 또는 숫자를 천간 문자로 치환하는 경우;

(d) 동일 문서 내에서 하나 이상의 천간 문자가 정상 인식된 경우, 이를 상호참조 기준으로 하여 다른 혼동 문자의 치환 확신도를 높이는 단계;

를 포함하는 것을 특징으로 한다.

이 방법의 핵심은, 동일 문서 내에서 하나라도 정상 인식된 천간 문자가 있으면 천간 체계의 존재가 확정되므로, 나머지 혼동 문자들도 천간 문자일 확률이 급격히 높아진다는 상호참조 원리를 활용하는 것이다.

#### 제5 발명: 페이지 복잡도 사전 분류 기반 적응적 AI 처리 방법

본 발명에 따른 페이지 복잡도 사전 분류 및 적응적 처리 방법은:

(a) PDF 문서의 각 페이지에서 이미 추출된 벡터 객체(선, 사각형) 및 텍스트 블록의 메타데이터를 API 호출 없이 로컬에서 분석하여, 각 페이지에 복잡도 태그 0(텍스트 전용) 또는 1(복합)을 부여하는 사전 분류 단계로서:
  - 구조선(structural line)의 수가 4개 이상이고 사각형(rect)이 2개 이상인 경우: 표(TABLE)가 존재하는 것으로 판정하여 태그 1 부여;
  - 페이지 폭의 30% 이상, 높이의 20% 이상인 대형 사각형이 존재하는 경우: 그림(FIGURE)이 존재하는 것으로 판정하여 태그 1 부여;
  - 텍스트 블록들의 수평 중심점(center_x)이 페이지 중앙을 기준으로 좌측과 우측에 각각 2개 이상 분포하는 경우: 다단(MULTI_COLUMN) 레이아웃으로 판정하여 태그 1 부여;
  - 디지털 텍스트 블록이 0개인 경우: 스캔 이미지 전용 페이지(IMAGE_ONLY)로 판정하여 태그 1 부여;
  - 상기 어떤 조건에도 해당하지 않는 경우: 태그 0 부여;
  이 사전 분류는 페이지당 약 0.1밀리초의 처리 시간으로 수행되는 단계;

(b) 태그 0이 부여된 텍스트 전용 페이지에 대해, 로컬 OCR 엔진(예: Surya)을 사용하여 고품질 텍스트를 먼저 추출하는 단계;

(c) 상기 태그 0 페이지의 OCR 결과에 스타일 메타데이터(폰트 크기, 볼드 여부, 정렬 방향)를 부가하되:
  - 디지털 PDF의 경우: PDF 텍스트 레이어에서 추출한 정확한 폰트 크기(pt), 폰트 이름, 볼드/이탤릭 플래그, 정렬 정보를 그대로 사용하는 단계;
  - 스캔 PDF의 경우: OCR 결과의 바운딩 박스 높이에 0.75를 곱하여 폰트 크기를 추정하고, 바운딩 박스의 수평 중심점과 페이지 폭의 비율로 정렬을 추론하는 단계;

(d) 상기 태그 0 페이지에 대해, 이미지를 전송하지 않고 OCR 텍스트와 스타일 메타데이터만을 다중모달 비전 AI에 전송하여, 텍스트 교정 및 제목 분류를 수행하는 단계;

(e) 태그 1이 부여된 복합 페이지에 대해, 페이지 이미지와 복잡도 상세 힌트(TABLE, FIGURE, MULTI_COL, IMAGE_ONLY)를 다중모달 비전 AI에 전송하여, OCR, 표 구조 인식, 레이아웃 분석, 제목 분류, 텍스트 교정을 일괄 수행하는 단계;

(f) 태그 0 페이지와 태그 1 페이지의 처리 결과를 원래의 페이지 순서로 병합하는 단계;

를 포함하며, 태그 0 페이지는 이미지 토큰 없이 텍스트 토큰만으로 처리되므로 페이지당 입력 토큰이 약 7.5배 절감되고, 태그 1 페이지는 복잡도 상세 힌트에 의해 AI가 분석 대상을 사전에 파악하여 정확도가 향상되는 것을 특징으로 한다.

#### 제6 발명: 복수 분석 작업의 단일 AI 호출 통합 처리 방법

본 발명에 따른 통합 비전 처리 방법은:

(a) 종래 기술에서 각각 별도의 AI 호출로 수행되던 읽기 순서 보정, 제목 계층 분류, 텍스트 교정의 3가지 분석 작업을, 복잡도 태그에 따라 최적화된 단일 프롬프트로 통합하는 단계;

(b) 태그 0 페이지에 대해, 다음의 정보만으로 구성된 텍스트 전용 프롬프트를 생성하는 단계:
  - 각 텍스트 블록의 텍스트 내용
  - 바운딩 박스 좌표 (읽기 순서 결정용)
  - 폰트 크기(pt), 볼드 여부, 정렬 방향 (제목 분류용)
  - 한국어 제목 패턴 분류 규칙 ("제1장"/"제1편" = h2, "제1절"/"제1관" = h3, "제1조" = h4)
  이 프롬프트에는 이미지가 포함되지 않으므로, 대형 배치(최대 30페이지)로 일괄 처리가 가능한 단계;

(c) 태그 1 페이지에 대해, 다음의 정보로 구성된 다중모달 프롬프트를 생성하는 단계:
  - 각 페이지의 이미지
  - 사전 분류에서 감지된 복잡도 상세 힌트 ([TAG=1] TABLE, FIGURE 등)
  - 기존 OCR/디지털 텍스트 힌트 (보정 참조용)
  - 표 구조 인식을 위한 셀 구조 추출 지시 (행, 열, rowspan, colspan)
  이 프롬프트에는 이미지가 포함되므로, 소형 배치(최대 10페이지)로 처리하는 단계;

(d) 상기 단일 AI 호출의 응답으로부터, 각 페이지별 텍스트 블록(교정된 텍스트, 바운딩 박스, 읽기 순서, 제목 레벨, 스타일 정보)과 표 구조(셀 배열)를 일괄 추출하는 단계;

를 포함하며, 종래 기술 대비 AI API 호출 횟수가 페이지당 3회에서 1회로 67% 감소하고, 태그 0 페이지의 입력 토큰이 88% 감소하여, 전체적으로 약 57%의 비용 절감 효과를 달성하는 것을 특징으로 한다.

#### 제7 발명: AI 변환-사전 교정-인간 교정의 3단계 품질 보증 방법

본 발명에 따른 문서 변환 품질 보증 방법은:

(a) 상기 제5 발명 및 제6 발명에 의한 AI 기반 자동 변환을 수행하여, 원본 PDF 문서의 텍스트, 표, 레이아웃, 제목 계층을 포함하는 1차 변환 결과를 생성하는 단계;

(b) 상기 1차 변환 결과에 대해, 사전(dictionary) 기반 자동 교정을 적용하는 단계로서:
  - 한자 천간(甲乙丙丁) 문자의 문맥 기반 보정 (상기 제4 발명)
  - 기호 교정 (OCR에서 빈번하게 오인식되는 특수문자 치환)
  - 사전 등재 어휘 기반 교정 (오탈자 치환 테이블)
  이 사전 기반 교정은 AI 호출 없이 로컬에서 수행되므로 추가 비용이 발생하지 않는 안전망(safety net) 역할을 하는 단계;

(c) 상기 사전 교정 결과를 HTML 및 마크다운 형식으로 자동 렌더링하여 최종 결과물을 출력하는 단계로서, 모든 AI 처리와 사전 교정이 완전히 완료된 후에 결과물이 사용자에게 제공되는 단계;

(d) 사용자가 필요시, 웹 기반 HTML 에디터에서 원본 PDF 이미지와 변환 결과를 좌우로 비교하면서, AI 및 사전 교정에서 미처 잡지 못한 잔여 오류(약 1~5%)를 수동으로 교정하는 단계;

(e) 사용자가 교정된 HTML을 저장한 후, 마크다운 재변환을 요청하면 교정된 내용이 반영된 마크다운 문서를 재생성하는 단계;

를 포함하며, AI 자동 변환(~95% 정확도) + 사전 자동 교정(~98% 정확도) + 인간 수동 교정(~100% 정확도)의 3단계 보완 구조에 의해, 99.99% 이상의 변환 정확도를 달성하는 것을 특징으로 한다.

이 방법의 핵심 원리는, AI가 대량의 반복 작업(수천 페이지의 텍스트 추출, 표 구조 인식, 레이아웃 분석)을 고속으로 처리하고, 사전 교정이 알려진 패턴의 오류를 일괄 보정하며, 인간이 최종적으로 문맥적 판단이 필요한 소수의 경계 사례를 확인·수정하는 "AI 대량 처리 → 규칙 기반 보정 → 인간 최종 검증"의 협업 구조이다.

#### 제8 발명: 문서 변환과 동시 번역의 단일 AI 호출 통합 방법

본 발명에 따른 문서 변환·동시 번역 방법은:

(a) 사용자가 문서 변환을 요청할 때, 변환과 동시에 번역을 수행할 것인지, 수행한다면 소스 언어(자동 감지 가능)와 타겟 언어를 선택하는 단계;

(b) 번역 옵션이 선택된 경우, 상기 제5 발명에 따른 페이지 복잡도 사전 분류(TAG=0/1)를 수행하되, TAG=0 페이지에 대해서도 로컬 OCR 엔진으로 텍스트를 추출한 후 해당 텍스트를 반드시 다중모달 비전 AI에 전송하는 단계로서:
  - TAG=0 페이지: 로컬 OCR로 텍스트를 추출한 후, 추출된 텍스트만(이미지 없이) Gemini에 전송하여 텍스트 교정 + 제목 분류 + 번역을 동시에 수행;
  - TAG=1 페이지: 페이지 이미지와 함께 Gemini에 전송하여 OCR + 표 구조 인식 + 레이아웃 분석 + 제목 분류 + 텍스트 교정 + 번역을 동시에 수행;
  두 경우 모두, 번역 지시는 기존의 변환용 프롬프트에 추가 문장으로 삽입되며, 별도의 AI 호출이 아닌 동일한 단일 호출 내에서 변환과 번역이 동시에 수행되는 단계;

(c) AI의 응답에서 번역된 텍스트가 각 블록의 text 필드에 직접 포함되어 반환되는 단계로서, 별도의 번역 후처리가 불필요한 단계;

(d) 번역된 텍스트를 포함한 블록들을 원래의 페이지 순서로 병합하고, HTML 및 마크다운 형식으로 렌더링하는 단계;

를 포함하며, 다음의 특징을 갖는다:

**비용 효율성**: 번역 지시는 기존 프롬프트에 2~3문장만 추가하는 것이므로 입력 토큰 증가는 약 5%에 불과하다. 출력 토큰은 번역된 텍스트의 길이에 따라 약 30% 증가한다. 따라서 전체 비용은 변환 전용 대비 약 25~35%만 증가하며, 이는 별도의 번역 API를 호출하는 종래 방식(변환 비용 + 번역 비용 = 약 200% 증가) 대비 현저히 저렴하다.

**정확도 우위**: AI가 문서의 시각적 레이아웃(표 구조, 제목 계층, 문단 관계)을 이해한 상태에서 번역을 수행하므로, 문서 구조를 모르는 상태에서 텍스트만 번역하는 종래의 기계번역(Google Translate, DeepL 등) 대비 문맥적 번역 정확도가 높다. 특히, 표 내부의 셀 텍스트, 각주와 본문의 참조 관계, 법률 용어의 문맥적 의미 등에서 구조 인식 기반 번역이 우수한 결과를 보인다.

**워크플로우 단순화**: 사용자는 문서를 업로드할 때 "번역" 체크박스와 언어를 선택하기만 하면 되며, 변환 결과물이 이미 번역된 상태로 출력된다. 별도의 번역 단계를 거칠 필요가 없으므로, 외국어 문서를 한국어 구조화 문서로 변환하는 전체 워크플로우가 단일 단계로 축약된다.

### [발명의 효과]

본 발명에 따르면, 다음과 같은 효과가 달성된다.

첫째, 구조선 기반 읽기 영역 분할에 의해, 다단 레이아웃, 표, 사이드바, 각주가 혼합된 복잡한 페이지에서도 인간이 읽는 순서와 동일한 읽기 순서를 결정할 수 있다. 인간이 문서에 선을 긋는 목적 자체가 읽기 순서를 지시하기 위한 것이므로, 이 선을 읽기 순서 결정의 기준으로 사용하는 것은 XY 좌표 클러스터링보다 본질적으로 정확하다.

둘째, 디지털 PDF 하이브리드 처리에 의해, 표 내부 텍스트의 뒤섞임 문제가 근본적으로 해결된다. 표의 외곽선이 존재하는 영역은 이미지 기반 AI가 표 구조를 인식하여 셀 단위로 정확하게 텍스트를 배치하고, 외곽선이 없는 텍스트 영역은 고속의 디지털 텍스트 추출을 사용하므로, 정확성과 처리 속도를 동시에 달성한다.

셋째, 벡터 라인 분류에 의해, 구조적 의미를 갖는 선(표 테두리, 구분선)과 설명적 의미를 갖는 선(화살표, 지시선)이 자동으로 구분되므로, 문서의 모든 시각적 요소가 의미적으로 정확하게 변환된다.

넷째, 문맥 기반 한자 천간 보정에 의해, 한국어 법률 문서, 시험 문제지 등에서 빈번하게 발생하는 OCR 오인식이 자동으로 교정되어, 문서의 의미적 정확성이 크게 향상된다.

다섯째, 페이지 복잡도 사전 분류(TAG=0/1)에 의해, 텍스트 전용 페이지(통상 전체의 60~80%)에 대해서는 이미지를 AI에 전송하지 않고 텍스트만 전송하므로, 입력 토큰이 페이지당 약 7.5배 절감된다. 2,000페이지 문서 기준으로 종래 약 $17.42의 API 비용이 약 $7.47로 57% 절감되며, 사전 분류 자체는 페이지당 0.1밀리초로 처리되어 추가 지연이 사실상 없다.

여섯째, 복수 분석 작업의 단일 AI 호출 통합에 의해, 종래 페이지당 3회의 AI 호출(읽기 순서 + 제목 분류 + 텍스트 교정)이 1회로 감소한다. 이는 API 호출 오버헤드를 67% 감소시키고, 동일 컨텍스트 내에서 복수 작업을 수행하므로 작업 간 일관성도 향상된다. 특히, 태그 0 페이지의 스타일 메타데이터(폰트 크기, 볼드, 정렬)를 AI에 제공함으로써, 이미지 없이도 제목 분류 정확도 95% 이상을 달성한다.

일곱째, 태그 기반 배치 크기 최적화에 의해, 텍스트 전용 페이지는 최대 30페이지, 복합 페이지는 최대 10페이지 단위로 배치 처리하여, AI의 컨텍스트 윈도우를 최대한 활용하면서도 복합 페이지에 충분한 출력 토큰을 할당한다.

여덟째, AI 자동 변환 → 사전 자동 교정 → 인간 수동 교정의 3단계 품질 보증 구조에 의해, 99.99% 이상의 변환 정확도를 달성한다. AI가 수천 페이지의 대량 작업을 고속으로 처리하고(~95%), 사전 교정이 알려진 패턴의 오류를 일괄 보정하며(~98%), 인간이 HTML 에디터에서 원본 PDF와 비교하면서 잔여 1~2%의 경계 사례를 최종 확인·수정한다. 이 협업 구조는 완전 자동화의 한계(미묘한 문맥 판단, 드문 레이아웃 패턴)를 인간의 판단력으로 보완하면서도, 인간의 노동량을 전체 페이지의 1~5%로 최소화한다.

아홉째, 문서 변환과 동시 번역의 단일 AI 호출 통합에 의해, 외국어 PDF 문서를 번역된 구조화 문서(HTML/마크다운)로 변환하는 전체 워크플로우가 단일 단계로 축약된다. 번역 지시가 기존 변환 프롬프트에 통합되므로, 별도의 번역 API 호출이 불필요하고, 추가 비용은 출력 토큰 약 30% 증가(변환 전용 대비 약 25~35% 비용 증가)에 불과하다. 이는 별도 번역 서비스를 사용하는 종래 방식(변환 + 번역 = 약 200% 비용) 대비 현저히 저렴하다. 또한, AI가 문서의 구조(표, 제목, 각주)를 이해한 상태에서 번역하므로, 구조를 모르는 텍스트 전용 번역 대비 문맥적 정확도가 높다.

### 실시예 2: 구조선 기반 읽기 영역 분할의 구체적 실시

도 1은 본 발명에 따른 구조선 기반 읽기 영역 분할의 처리 흐름을 나타낸다.

#### 단계 S100: 벡터 라인 추출

PDF 페이지의 렌더링 명령어를 파싱하여, 모든 직선(line)과 사각형(rectangle)의 좌표를 추출한다. PyMuPDF 등의 라이브러리에서 제공하는 get_drawings() API를 사용하여, 각 드로잉 객체의 시작점(x0, y0)과 끝점(x1, y1)을 획득한다.

#### 단계 S110: 선 분류 (2단계 분류법)

선의 분류는 단일 선의 각도만으로는 완전하지 않다. 수평 또는 수직 방향의 선이라도 표의 테두리가 아닌 부연설명용 지시선일 수 있기 때문이다. 따라서 본 발명은 2단계 분류법을 사용한다.

**S111. 1차 분류 (각도 기반 예비 분류)**

각 벡터 라인에 대해 수평축 대비 각도 theta를 다음과 같이 계산한다:

```
dx = |x1 - x0|
dy = |y1 - y0|
length = sqrt(dx^2 + dy^2)
theta = atan2(dy, dx) * (180 / pi)
```

예비 분류 기준:
- theta <= 5도 또는 theta >= 85도: 잠정적 구조선 후보 (structural candidate)
- 5도 < theta < 85도: 확정 지시선 (annotation) — 대각선은 항상 지시선
- length < 5px: 점(dot)으로 간주하여 무시

**S112. 2차 분류 (구조적 참여 여부 판정)**

1차에서 "잠정적 구조선 후보"로 분류된 수평/수직 선에 대해, 해당 선이 닫힌 구조(격자, 사각형, 표 테두리)를 형성하는 데 참여하는지를 판정한다.

판정 기준: 해당 선의 **양쪽 끝점(endpoint) 중 적어도 하나**가 다음 조건 중 하나를 만족해야 구조선으로 확정된다:
- 수직 방향의 다른 구조선 후보와 소정 허용 범위(예: 15px) 이내에서 교차하는 경우 (수평선의 경우)
- 수평 방향의 다른 구조선 후보와 소정 허용 범위 이내에서 교차하는 경우 (수직선의 경우)
- 사각형(rect)의 꼭짓점과 소정 허용 범위 이내에 위치하는 경우

위 조건을 만족하지 않는 수평/수직 선은 **지시선(annotation)으로 재분류**된다. 이는 해당 선이 닫힌 영역을 형성하지 않고 한쪽 끝이 열린 채로 텍스트를 가리키는 부연설명용 선임을 의미한다.

핵심 원리: **구조선은 영역을 '닫는'(enclosed) 선**이고, **지시선은 영역을 '열어두는'(open-ended) 선**이다. 표의 테두리는 반드시 다른 선과 만나서 격자를 형성하지만, 부연설명용 지시선은 한쪽 끝이 여백이나 텍스트에서 끝난다.

**S113. 애매한 경우의 맥락 기반 판정**

2차 분류에서도 판별이 애매한 경우(예: 수평선의 한 끝점만 다른 선과 만나는 경우), 해당 선의 양쪽에 위치한 텍스트 블록의 의미적 관련성을 분석한다:
- 양쪽 텍스트가 서로 밀접하게 관련된 부연설명 관계인 경우 → 지시선
- 양쪽 텍스트가 서로 독립적인 내용(다른 섹션, 다른 셀)인 경우 → 구조선

이 맥락 판정은 VLM(시각-언어 모델) 또는 LLM(대규모 언어 모델)에 해당 선 주변의 이미지 또는 텍스트를 제공하여 수행한다.

#### 단계 S120: 읽기 영역 생성

구조선과 사각형으로부터 읽기 영역을 생성하는 알고리즘:

**S121. 사각형 기반 영역 생성**
- 폭 15px 이상, 높이 10px 이상의 사각형을 각각 하나의 읽기 영역으로 설정
- 상호 30% 이상 중첩되는 사각형은 병합하여 하나의 영역으로 통합

**S122. 수평 구분선 기반 분할**
- 페이지 폭의 15% 이상 길이의 수평 구조선의 Y좌표를 수집
- 근접한 Y좌표(차이 15px 이내)를 중복 제거
- 수집된 Y좌표와 페이지 상단(Y=0) 및 하단(Y=page_height)을 결합하여 수평 밴드 생성

**S123. 수직 구분선 기반 분할**
- 페이지 높이의 10% 이상 길이의 수직 구조선의 X좌표를 수집
- 근접한 X좌표(차이 15px 이내)를 중복 제거
- 수집된 X좌표와 페이지 좌측(X=0) 및 우측(X=page_width)을 결합하여 각 수평 밴드를 수직으로 분할

**S124. 격자 영역 생성**
- 수평 밴드와 수직 분할의 교차점으로 격자(grid) 형태의 읽기 영역 생성
- 폭 또는 높이가 20px 미만인 영역은 제거
- 사각형 기반 영역과 70% 이상 중첩되는 격자 영역은 제거 (중복 방지)

#### 단계 S130: 블록-영역 할당

각 텍스트 블록의 바운딩 박스와 각 읽기 영역의 바운딩 박스 간 중첩 비율(overlap ratio)을 계산하여, 가장 높은 중첩 비율(단, 15% 이상)을 갖는 영역에 블록을 할당한다. 어떤 영역에도 15% 이상 중첩되지 않는 블록은 "미할당 블록"으로 분류하여 마지막에 배치한다.

#### 단계 S140: 영역 유형 분류 및 정렬

- 페이지 높이의 75% 이하에 위치한 영역: 각주 영역으로 분류
- 그 외 영역: 콘텐츠 영역으로 분류
- 정렬 순서: 콘텐츠 영역 (Y좌표 오름차순, 동일 Y에서 X좌표 오름차순) -> 각주 영역 -> 미할당 블록

### 실시예 3: 디지털 PDF 하이브리드 처리의 구체적 실시

도 2는 본 발명에 따른 디지털 PDF 하이브리드 처리의 흐름을 나타낸다.

#### 단계 S200: 디지털 텍스트 및 벡터 라인 동시 추출

디지털 PDF의 각 페이지에서 (1) 내장된 텍스트 블록과 (2) 벡터 라인/사각형을 동시에 추출한다. 디지털 텍스트 블록은 각각 좌표, 폰트 정보, 스타일 정보를 포함한다.

#### 단계 S210: 구조선 영역 탐지

추출된 벡터 라인 중 구조선으로 분류된 것들을 이용하여, 다음과 같은 구조선 영역을 탐지한다:

**S211. 표 영역 탐지**
- 수평 구조선과 수직 구조선이 격자(grid) 패턴을 형성하는 영역
- 사각형이 밀집된 영역 (표의 셀 테두리)
- 페이지 폭의 20% 이상, 높이의 5% 이상인 대형 사각형

**S212. 박스 영역 탐지**
- 단일 사각형으로 둘러싸인 텍스트 영역 (예: 정보 박스, 경고 박스)
- 표 영역과 70% 이상 중첩되지 않는 독립적 사각형

#### 단계 S220: 영역 분리 처리

```
만약 (디지털 텍스트 존재 AND 구조선 영역 탐지됨):
    1. 디지털 텍스트 블록 중 구조선 영역과 50% 이상 중첩되는 블록 -> 제외
    2. 페이지 이미지를 이미지 기반 AI 파이프라인으로 처리:
       a. 레이아웃 감지 (Surya, YOLO 등)
       b. OCR (Tesseract, EasyOCR 등)
       c. 테이블 구조 인식 (Table Transformer 등)
    3. AI 결과 중 구조선 영역과 30% 이상 중첩되는 블록만 선별
    4. 선별된 AI 블록 + 제외되지 않은 디지털 텍스트 블록 = 최종 병합
만약 아니면:
    종래 방식으로 처리 (디지털 텍스트 추출 또는 전체 이미지 AI)
```

이 방법의 핵심 원리: 인간이 표의 외곽선을 그리는 목적은 "이 셀을 다 읽고 다음 셀로 이동하라"는 읽기 순서 지시이다. 디지털 텍스트 추출은 이 지시(외곽선)를 버리고 텍스트만 가져오므로 순서가 뒤섞인다. 반면 이미지 기반 AI는 외곽선을 시각적으로 인식하여 셀 구조를 복원하므로, 표 영역에는 이미지 AI를 적용해야 한다.

### 실시예 5: 한자 천간 문자 OCR 보정의 구체적 실시

#### 단계 S400: 천간 빈출 문서 판별

OCR 결과 텍스트 전체에서 다음의 신호를 탐지한다:

**문서 유형 키워드 신호:**
| 문서 유형 | 판별 키워드 |
|---------|----------|
| 시험 문제지 | 문항, 정답, 배점, 수험번호, 시험시간 |
| 법률 문서 | 계약, 원고, 피고, 판결, 조항, 제~조, 위반 |
| 관공서 양식 | 신청인, 피신청인, 등기, 관할, 처분 |

**상호참조 신호:**
동일 문서 내에서 甲, 乙, 丙, 丁, 戊 중 하나 이상이 정상 인식된 경우, "천간 체계 확정"으로 판단한다. 이는 나머지 혼동 문자가 천간 문자일 확률을 결정적으로 높이는 핵심 신호이다.

#### 단계 S410: 혼동 문자 치환

"천간 빈출 문서"로 판별된 경우, 다음의 치환 규칙을 적용한다:

| OCR 오인식 | 올바른 문자 | 치환 조건 |
|-----------|-----------|---------|
| Z | 乙(을) | 주어 위치, 조사(은/는/이/가/에게) 앞 |
| T | 丁(정) | 주어 위치, 조사 앞 |
| 2 | 乙(을) | 단독 숫자로 조사 앞에 출현 |
| 7 | 丁(정) | 단독 숫자로 조사 앞에 출현 |
| 田 또는 由 | 甲(갑) | 주어 위치 |
| 内 | 丙(병) | 주어 위치 |

치환 시 확신도(confidence) 산출:
```
base_confidence = 0.5  (문서 유형 키워드 매칭 시)
if 동일_문서_내_정상_천간_존재:
    confidence += 0.35  (상호참조 보너스)
if 조사_패턴_매칭("은/는/이/가/에게"):
    confidence += 0.10
if 순서_패턴_매칭(갑->을->병->정->무):
    confidence += 0.05
```

confidence >= 0.8인 경우에만 치환을 실행한다.

### 실시예 6: 표 내부 읽기 순서의 구체적 실시

표(테이블) 내부의 읽기 순서는 외곽선(셀 테두리)에 의해 결정된다:

#### 단계 S500: 셀 구조 인식

이미지 기반 AI(Table Transformer 등) 또는 OpenCV 기반 라인 감지를 통해, 표의 수평선과 수직선 위치를 탐지하고, 이들의 교차점으로 셀 격자를 구성한다.

#### 단계 S510: 셀 내부 텍스트 읽기

각 셀 내부의 텍스트를 위에서 아래로, 왼쪽에서 오른쪽으로 읽는다. 핵심은 하나의 셀 내부의 텍스트를 완전히 읽은 후 다음 셀로 이동하는 것이다. 이것이 인간이 표의 외곽선으로 독자에게 지시하는 읽기 순서이다.

#### 단계 S520: 셀 간 이동 순서

같은 행(row) 내에서 왼쪽 셀 -> 오른쪽 셀로 이동한 후, 다음 행으로 이동한다. 셀 병합(rowspan/colspan)이 있는 경우, 병합된 셀은 해당 셀이 시작되는 위치(최소 row, 최소 col)에서 한 번만 읽는다.

### 실시예 7: 이미지 PDF(스캔 문서) 처리의 구체적 실시

이미지 PDF의 경우, 모든 내용이 래스터 이미지이므로 전체 페이지에 대해 이미지 기반 AI 파이프라인을 적용한다:

1. 페이지를 고해상도 이미지(300 DPI 이상)로 렌더링
2. 레이아웃 감지 모델로 텍스트 영역, 표 영역, 그림 영역 등을 식별
3. OpenCV로 벡터 라인에 해당하는 직선을 감지 (Hough Transform 등)
4. 감지된 직선을 구조선/지시선으로 분류 (상기 각도 기반 분류)
5. 구조선 기반 읽기 영역 분할 (상기 실시예 2와 동일)
6. 각 블록에 대해 OCR 수행
7. 한자 천간 등 문맥 기반 OCR 보정 수행
8. HTML/마크다운으로 변환

### 실시예 8: 페이지 복잡도 사전 분류의 구체적 실시

도 10은 본 발명의 제5 실시예에 따른 페이지 복잡도 사전 분류 방법의 처리 흐름을 나타낸다.

#### 단계 S600: 벡터 객체 메타데이터 수집

PDF 페이지에서 이미 추출된 벡터 라인 및 사각형의 좌표 정보를 수집한다. 이 정보는 페이지 렌더링 단계(단계 S100)에서 이미 추출된 것이므로 추가적인 파싱 비용이 발생하지 않는다.

#### 단계 S610: 복잡도 판정 규칙 적용

각 페이지에 대해 다음의 규칙을 순차적으로 적용하여, 하나라도 해당하면 즉시 TAG=1(복합)로 분류하고, 어느 것에도 해당하지 않으면 TAG=0(텍스트 전용)으로 분류한다.

**S611. 표(TABLE) 존재 판정**
```
만약 (구조선 수 >= 4) AND (사각형 수 >= 2):
    TAG = 1, 상세힌트 = "TABLE"
아니면 만약 (사각형 수 >= 4):
    TAG = 1, 상세힌트 = "TABLE"
```
표는 최소 4개의 구조선(수평 2개 + 수직 2개)과 2개의 사각형(셀 영역)으로 구성되는 것이 통상적이므로, 이 임계값은 거의 모든 표를 감지한다.

**S612. 그림(FIGURE) 존재 판정**
```
각 사각형에 대해:
    만약 (사각형 폭 > 페이지 폭 * 0.3) AND (사각형 높이 > 페이지 폭 * 0.2):
        TAG = 1, 상세힌트 = "FIGURE"
```
페이지 폭의 30% 이상을 차지하는 대형 사각형은 그림, 차트, 또는 수식 영역의 플레이스홀더일 가능성이 높다.

**S613. 다단(MULTI_COLUMN) 레이아웃 판정**
```
만약 (텍스트 블록 수 >= 4):
    mid = 페이지 폭 / 2
    left_count = 중심점 X < mid * 0.85 이고 폭 < 페이지 폭 * 0.6인 블록 수
    right_count = 중심점 X > mid * 1.15 이고 폭 < 페이지 폭 * 0.6인 블록 수
    만약 (left_count >= 2) AND (right_count >= 2):
        TAG = 1, 상세힌트 = "MULTI_COL"
```
텍스트 블록이 페이지 좌반과 우반에 각각 2개 이상 분포하면 2단 이상의 다단 레이아웃으로 판정한다. 각 블록의 폭이 페이지 폭의 60% 미만이어야 한다는 조건은, 단일 행이 전체 폭을 차지하는 제목/머리글을 다단으로 오판하는 것을 방지한다.

**S614. 스캔 이미지 전용 페이지(IMAGE_ONLY) 판정**
```
만약 (디지털 텍스트 블록 수 == 0):
    TAG = 1, 상세힌트 = "IMAGE_ONLY"
```
디지털 텍스트가 전혀 없는 페이지는 전체가 스캔 이미지이므로, AI가 이미지에서 직접 OCR을 수행해야 한다.

#### 단계 S620: 태그 기반 배치 분리

```
text_only_pages = [TAG=0인 페이지들]
complex_pages = [TAG=1인 페이지들]
```

TAG=0 페이지는 최대 30페이지 단위의 대형 배치로, TAG=1 페이지는 최대 10페이지 단위의 소형 배치로 분리한다. 이 비대칭 배치 크기의 핵심 원리는:
- TAG=0 배치: 이미지 없이 텍스트만 전송하므로 입력 토큰이 적고, AI의 출력도 교정/제목 분류에 한정되므로 출력 토큰도 적어, 대형 배치가 가능
- TAG=1 배치: 페이지당 이미지 토큰(~1,500)이 대량으로 소비되고, 표 구조 등 상세 출력이 필요하므로, 소형 배치로 제한

#### 단계 S630: TAG=0 페이지의 로컬 OCR 선처리

TAG=0으로 분류된 페이지 중 디지털 텍스트 레이어가 없는 페이지(스캔되었으나 표/그림이 없는 텍스트 전용 스캔 페이지)에 대해, 로컬 OCR 엔진(Surya)을 실행하여 텍스트를 추출한다.

디지털 텍스트 레이어가 존재하는 PDF 페이지에 대해서는, PDF 텍스트 레이어에서 추출한 텍스트 블록을 그대로 사용한다. 이 경우 폰트 크기, 볼드 여부, 이탤릭 여부, 정렬 방향, 색상 등의 정확한 스타일 정보가 함께 제공된다.

#### 단계 S640: 스타일 메타데이터 부가 및 AI 전송

TAG=0 페이지의 각 텍스트 블록에 다음의 스타일 메타데이터를 부가하여 AI에 전달한다:

| 정보 항목 | 디지털 PDF | 스캔 PDF |
|---------|-----------|---------|
| 폰트 크기 | PDF 레이어에서 정확한 pt 값 | bbox 높이 × 0.75로 추정 |
| 볼드 여부 | 폰트 플래그 또는 폰트명에서 감지 | 감지 불가 (AI가 텍스트 패턴으로 추론) |
| 정렬 | bbox 중심점과 페이지 폭 비율로 계산 | bbox 중심점과 페이지 폭 비율로 계산 |
| 이탤릭 | 폰트 플래그에서 감지 | 감지 불가 |

AI에 전달되는 형식 예시:
```
Page 5  [TAG=0] (2550x3300px):
  OCR text (verify/correct):
  - "제1장 총칙" bbox=[320,150,2230,210] (sz=18.5pt, BOLD, align=center)
  - "제1조(목적) 이 법은..." bbox=[300,280,2250,320] (sz=12.0pt)
  - "② 이 법에서 사용하는..." bbox=[300,350,2250,390] (sz=12.0pt)
```

이 형식에 의해, AI는 이미지 없이도 다음을 판단할 수 있다:
- "제1장 총칙"은 sz=18.5pt + BOLD + align=center이므로 h2 제목
- "제1조(목적)"는 한국어 조항 패턴 "제N조"이므로 h4 제목
- "② 이 법에서..."는 sz=12.0pt 일반 텍스트이므로 본문(paragraph)

### 실시예 9: 통합 비전 처리 및 비용 최적화의 구체적 실시

도 11은 본 발명의 제6 실시예에 따른 통합 비전 처리 방법의 처리 흐름을 나타낸다.

#### 단계 S700: 종래 기술의 다중 AI 호출 방식 (비교 대상)

종래 기술에서는 하나의 문서 청크(예: 10페이지)를 처리하기 위해 다음의 3회 AI 호출이 필요하였다:

```
[출력 형식]
  JSON: {pages: [{page_index, blocks: [{text, bbox, reading_order, heading_level, style, table}]}]}
```

#### 단계 S720: 비용 절감 효과의 정량적 분석

2,000페이지 문서(TAG=0: 1,400페이지(70%), TAG=1: 600페이지(30%))를 Gemini 3.1 Flash-Lite (입력 $0.50/1M 토큰, 출력 $3.00/1M 토큰)으로 처리할 경우:

**종래 방식 (모든 페이지에 이미지 전송 + 3회 호출)**

| 항목 | 계산 | 금액 |
|------|------|------|
| 입력 토큰 | 2,000 × 1,650 = 3,300,000 | $1.65 |
| 출력 토큰 | 2,000 × 1,200 = 2,400,000 | $7.20 |
| **합계 (raw)** | | **$8.85** |
| **합계 (2.2배 마크업)** | | **$19.47** |

**본 발명 방식 (TAG=0에 텍스트만 전송 + 단일 호출)**

| 항목 | TAG=0 (1,400p) | TAG=1 (600p) | 합계 |
|------|----------------|--------------|------|
| 입력 토큰 | 1,400 × 200 = 280,000 | 600 × 1,650 = 990,000 | 1,270,000 |
| 출력 토큰 | 1,400 × 600 = 840,000 | 600 × 1,500 = 900,000 | 1,740,000 |
| 입력 비용 | $0.14 | $0.50 | **$0.64** |
| 출력 비용 | $2.52 | $2.70 | **$5.22** |
| **합계 (raw)** | | | **$5.86** |
| **합계 (2.2배 마크업)** | | | **$12.89** |

**절감율: ($19.47 - $12.89) / $19.47 = 약 34% 절감**

#### 단계 S730: 속도 최적화

배치 처리의 병렬화에 의한 속도 향상:

| 항목 | TAG=0 | TAG=1 |
|------|-------|-------|
| 배치 크기 | 30페이지 | 10페이지 |
| 배치 수 | 1,400 / 30 = 47개 | 600 / 10 = 60개 |
| 최대 병렬 워커 수 | 8 | 8 |
| 예상 배치당 처리 시간 | ~2초 (텍스트만) | ~5초 (이미지 포함) |
| 병렬 처리 시 총 시간 | 47 / 8 × 2 ≈ 12초 | 60 / 8 × 5 ≈ 38초 |
| **총 AI 처리 시간** | | **약 50초** |

종래 방식(3회 호출 × 200개 청크 = 600회)과 비교하여, 본 발명은 총 47 + 60 = 107회의 AI 호출로 동일한 결과를 달성하며, 병렬화에 의해 2,000페이지를 약 50초 내에 처리할 수 있다.

### 실시예 10: AI-사전-인간 3단계 품질 보증의 구체적 실시

도 13은 본 발명의 제7 실시예에 따른 3단계 품질 보증 방법의 처리 흐름을 나타낸다.

#### 단계 S800: AI 자동 변환 (1단계)

상기 실시예 8 및 9에 따른 적응적 AI 처리에 의해, PDF 문서의 모든 페이지에 대해 텍스트 추출, 표 구조 인식, 레이아웃 분석, 제목 분류, 읽기 순서 결정, 텍스트 교정을 자동으로 수행한다. 이 단계의 정확도는 통상 93~97% 수준이다. 잔여 오류의 유형은:
- 미묘한 폰트 크기 차이에 의한 제목 레벨 오분류 (예: h3을 h4로)
- 드문 레이아웃 패턴(원형 배치, 대각선 텍스트)의 읽기 순서 오류
- 전문 용어, 고유명사, 약어의 OCR 오인식 (사전 미등재 어휘)

#### 단계 S810: 사전 기반 자동 교정 (2단계)

AI 변환 결과에 대해, AI 호출 없이 로컬에서 다음의 규칙 기반 교정을 일괄 적용한다:

**S811. 한자 천간 교정** (상기 제4 발명)
- 문서 유형 판별 → 상호참조 → 혼동 문자 치환

**S812. 기호 교정**
- OCR에서 빈번하게 오인식되는 특수문자를 일괄 치환
- 예: "~" → "∼", "|" → "│", "O" (영문 대문자) → "0" (숫자, 문맥에 따라)

**S813. 사전 등재 어휘 교정**
- 오탈자 치환 테이블(correction_dict.json)에 등재된 패턴을 일괄 적용
- 예: "재판매가격유지행위" → 올바른 형태 확인 및 교정

이 2단계 교정에 의해 정확도가 약 98~99%로 향상된다. 이 단계는 AI 호출 비용이 발생하지 않으며, 2,000페이지에 대해 1초 미만으로 처리된다.

#### 단계 S820: 인간 수동 교정 (3단계)

**S821. 자동 처리 완료 후 최종 결과물 출력**

1단계(AI 자동 변환)와 2단계(사전 자동 교정)는 인간의 개입 없이 완전 자동으로 연속 수행된다. 모든 자동 처리가 완료되면, 최종 HTML 파일과 마크다운 파일이 자동으로 생성되어 사용자에게 제공된다. 이 시점에서 사용자는 이미 95~99% 정확도의 변환 결과를 보유하고 있으며, 추가 교정 없이도 바로 사용할 수 있다.

**S822. 사용자 리뷰 및 수동 교정 (선택적)**

사용자가 더 높은 정확도를 원하는 경우, 웹 기반 WYSIWYG HTML 에디터에서 변환 결과를 리뷰한다. 인터페이스는 다음의 기능을 제공한다:

```
+----------------------------+----------------------------+
|     [원본 PDF 이미지]        |     [변환 결과 HTML]        |
|                            |                            |
|  (페이지 단위로 표시)         |  (편집 가능한 HTML 에디터)    |
|                            |                            |
|  확대/축소, 페이지 이동      |  텍스트 직접 편집            |
|                            |  제목 레벨 변경 (드롭다운)    |
|                            |  표 셀 편집                 |
|                            |  블록 순서 드래그 이동        |
+----------------------------+----------------------------+
|  [이전 페이지] [다음 페이지]  |  [HTML 저장] [마크다운 변환]  |
+----------------------------+----------------------------+
```

사용자는 좌측의 원본 PDF 이미지와 우측의 변환 결과를 비교하면서, 필요한 경우에만 다음의 잔여 오류를 수정한다:
- 제목 레벨 오분류: 드롭다운에서 올바른 레벨(h1~h6) 선택
- 텍스트 오류: 변환 결과의 텍스트를 직접 클릭하여 편집
- 표 셀 내용 오류: 셀을 클릭하여 내용 수정
- 읽기 순서 오류: 블록을 드래그하여 순서 변경
- 누락된 내용: 원본에는 있으나 변환 결과에 없는 내용 추가

통상적인 문서에서 인간이 수정해야 하는 페이지는 전체의 1~5%이므로, 2,000페이지 문서에서 20~100페이지만 확인·수정하면 된다.

**S823. 교정 반영 및 재변환**

사용자가 HTML 에디터에서 교정을 마친 후:
1. [HTML 저장] 버튼을 클릭하면 교정된 내용이 HTML 파일로 저장된다.
2. [마크다운 변환] 버튼을 클릭하면, 교정된 HTML을 기반으로 마크다운 파일이 재생성된다. 이때 사용자가 수정한 내용(텍스트 교정, 제목 레벨 변경, 표 구조 수정 등)이 마크다운에 정확히 반영된다.

이 워크플로우의 핵심은, 모든 AI 처리가 완전히 끝난 후에 결과물이 나오고, 인간 교정은 이후 필요시에만 수행하는 비동기 구조라는 것이다. 인간의 느린 교정 속도가 AI의 고속 처리를 블로킹하지 않으며, 사용자는 교정 없이도 결과를 바로 사용하거나, 원하는 만큼만 교정한 후 재변환할 수 있다.

#### 3단계 품질 보증의 정확도 누적 분석

| 단계 | 처리 주체 | 정확도 | 잔여 오류율 | 처리 비용 |
|------|----------|--------|-----------|---------|
| 1단계: AI 자동 변환 | Gemini 3.1 Flash-Lite | ~95% | ~5% | API 토큰 비용 |
| 2단계: 사전 자동 교정 | 로컬 규칙 엔진 | ~98% | ~2% | 무비용 (로컬) |
| 3단계: 인간 수동 교정 | 사용자 (HTML 에디터) | ~100% | ~0% | 인간 노동 (1~5% 페이지만) |
| **최종** | | **99.99%+** | **<0.01%** | |

이 구조의 핵심은, AI가 전체 노동의 95%를 자동으로 처리하고, 사전 교정이 추가 3%를 무비용으로 처리하며, 인간은 최종 1~2%의 경계 사례에만 집중하는 "역할 분담"이다. 이에 의해, 2,000페이지 분량의 대형 문서를 인간이 처음부터 수동으로 교정하는 경우 수일이 소요되는 작업이, 본 발명에 의해 수십 분 이내로 단축된다.

### 실시예 11: 문서 변환과 동시 번역의 구체적 실시

도 14는 본 발명의 제8 실시예에 따른 문서 변환·동시 번역 방법의 처리 흐름을 나타낸다.

#### 단계 S900: 사용자의 번역 옵션 선택

사용자가 PDF 문서를 업로드할 때, 변환 옵션과 함께 다음을 선택한다:
- 번역 수행 여부 (체크박스)
- 소스 언어: 자동 감지(기본값), 또는 명시적 선택 (일본어, 영어, 중국어, 독일어, 프랑스어, 스페인어, 베트남어, 태국어, 러시아어, 포르투갈어, 이탈리아어, 아랍어, 인도네시아어 등)
- 타겟 언어: 한국어(기본값), 또는 기타 언어

#### 단계 S910: 복잡도 사전 분류 및 로컬 OCR 선처리

상기 실시예 8(단계 S600~S640)과 동일하게, 각 페이지에 TAG=0 또는 TAG=1을 부여한다.

TAG=0 페이지에 대해 로컬 OCR 엔진(Surya)으로 텍스트를 추출한다. 번역 모드에서도 TAG=0 페이지는 이미지를 AI에 전송하지 않고, 로컬 OCR로 추출된 텍스트만을 AI에 전송한다. 이는 번역 모드에서도 이미지 토큰 절감 효과를 유지하는 핵심이다.

#### 단계 S920: 번역 지시의 프롬프트 통합

기존의 변환용 AI 프롬프트에 번역 지시를 추가한다. 추가되는 지시는 2~3문장에 불과하므로 입력 토큰 증가는 미미하다(약 50토큰).

**TAG=0 배치의 번역 통합 프롬프트 예시:**
```
[기존 변환 지시 (텍스트 교정, 제목 분류, 읽기 순서)]

**TRANSLATION**: Translate ALL text from Japanese to Korean.
The 'text' field in your JSON output must contain the TRANSLATED text in Korean.
Preserve the original meaning accurately.
For proper nouns and technical terms, keep the original in parentheses,
e.g. '인공지능(AI)' or '동경도(東京都)'.
Table cell text must also be translated.

[페이지 데이터 (텍스트만, 이미지 없음)]
  Page 5 [TAG=0]: "第１章 総則" bbox=... (sz=18.5pt, BOLD, align=center)
  Page 5 [TAG=0]: "第１条（目的）この法律は..." bbox=... (sz=12.0pt)
```

**AI 응답 예시 (번역 완료):**
```json
{
  "pages": [{
    "page_index": 5,
    "blocks": [
      {
        "id": "p5_b0",
        "type": "heading",
        "text": "제1장 총칙(総則)",
        "heading_level": "h2",
        "reading_order": 0
      },
      {
        "id": "p5_b1",
        "type": "paragraph",
        "text": "제1조(목적) 이 법률은...",
        "heading_level": "h4",
        "reading_order": 1
      }
    ]
  }]
}
```

**TAG=1 배치의 번역 통합 프롬프트:**

TAG=1 페이지에서도 동일하게 번역 지시가 프롬프트에 추가된다. AI는 페이지 이미지에서 표 구조를 인식하면서 동시에 셀 내용을 번역한다.

```
[기존 변환 지시 (레이아웃 감지, 표 인식, OCR, 제목 분류)]

**TRANSLATION**: Translate ALL text from Japanese to Korean.
...동일한 번역 지시...

[페이지 이미지들]
  Page 12 [TAG=1] TABLE: ...
```

#### 단계 S930: 번역 결과의 직접 통합

AI의 응답에서 각 블록의 text 필드에는 이미 번역된 텍스트가 포함되어 있다. 따라서 별도의 번역 후처리 단계가 불필요하며, 기존의 HTML/마크다운 렌더링 파이프라인을 그대로 사용하여 번역된 구조화 문서를 생성한다.

#### 비용 비교: 변환 전용 vs 변환+번역 vs 종래 방식

2,000페이지 일본어 문서를 한국어로 변환+번역하는 경우 (Gemini 3.1 Flash-Lite):

| 방식 | 입력 토큰 | 출력 토큰 | Raw 비용 | 마크업 후 비용 |
|------|----------|----------|---------|-------------|
| 본 발명: 변환 전용 | 1,270,000 | 1,740,000 | $5.86 | $12.89 |
| 본 발명: 변환+번역 | 1,333,500 (+5%) | 2,262,000 (+30%) | $7.46 | $16.41 |
| 종래: 변환 후 별도 번역 API | 1,270,000 + 3,480,000 | 1,740,000 + 3,480,000 | $5.86 + $13.92 | $43.52 |

**본 발명의 변환+번역 비용은 종래 방식의 약 38%에 불과하며, 변환 전용 대비 약 27%만 추가된다.**

## [특허청구범위]

### [청구항 1] (독립항 - 구조선 기반 읽기 순서 결정 방법)

PDF 문서를 구조화된 문서로 변환하기 위한 컴퓨터 구현 방법으로서,

(a) PDF 문서의 각 페이지에서 벡터 라인 및 사각형 정보를 추출하는 단계;

(b) 상기 추출된 벡터 라인을 각 라인의 수평축 대비 각도에 기초하여, 수평 방향 또는 수직 방향의 구조선과 그 외 각도의 지시선으로 분류하는 단계;

(c) 상기 구조선 및 사각형을 이용하여, 상기 페이지를 복수의 비중첩 읽기 영역으로 분할하는 단계;

(d) 상기 페이지의 각 텍스트 블록을 해당 블록과 가장 높은 중첩 비율을 갖는 읽기 영역에 할당하는 단계; 및

(e) 각 읽기 영역 내에서 텍스트 블록을 위에서 아래로, 왼쪽에서 오른쪽으로 정렬하여 읽기 순서를 결정하는 단계;

를 포함하는 것을 특징으로 하는 방법.

### [청구항 2] (종속항)

청구항 1에 있어서,

상기 단계 (b)는:
(b-1) 수평축 대비 각도가 5도 이하 또는 85도 이상인 라인을 잠정적 구조선 후보로, 5도 초과 85도 미만인 라인을 지시선으로 예비 분류하는 단계; 및
(b-2) 상기 잠정적 구조선 후보 중, 양쪽 끝점이 모두 다른 구조선 후보 또는 사각형의 꼭짓점과 소정 허용 범위 이내에서 교차하지 않는 라인을 지시선으로 재분류하는 단계;
를 포함하는 2단계 분류를 수행하는 것을 특징으로 하는 방법.

### [청구항 3] (종속항)

청구항 1에 있어서,

상기 단계 (c)는:
(c-1) 폭 및 높이가 소정 임계값 이상인 사각형을 각각 하나의 읽기 영역으로 설정하는 단계;
(c-2) 페이지 폭의 소정 비율 이상 길이의 수평 구조선의 Y좌표로 페이지를 수평 밴드로 분할하는 단계; 및
(c-3) 페이지 높이의 소정 비율 이상 길이의 수직 구조선의 X좌표로 상기 수평 밴드를 수직으로 분할하여 격자 형태의 읽기 영역을 생성하는 단계;
를 포함하는 것을 특징으로 하는 방법.

PDF 문서를 구조화된 문서로 변환하는 과정에서 변환 품질을 보증하기 위한 컴퓨터 구현 방법으로서,

(a) 다중모달 비전 AI를 이용하여 PDF 문서의 각 페이지에 대해 텍스트 추출, 레이아웃 분석, 제목 분류, 및 텍스트 교정을 자동으로 수행하여 1차 변환 결과를 생성하는 단계;

(b) 상기 1차 변환 결과에 대해, AI 호출 없이 로컬에서 사전 기반 자동 교정을 적용하는 단계로서, 한자 천간 문자의 문맥 기반 보정, 기호 교정, 및 사전 등재 어휘 기반 교정 중 하나 이상을 포함하는 단계;

(c) 상기 사전 교정 결과를 HTML 및 마크다운 형식으로 자동 렌더링하여, 모든 자동 처리가 완료된 최종 결과물을 사용자에게 제공하는 단계;

(d) 사용자가 필요시, 원본 PDF 이미지와 변환 결과를 비교 가능한 HTML 에디터 인터페이스에서 잔여 오류를 수동으로 교정하는 단계; 및

(e) 사용자가 교정된 HTML을 저장한 후 마크다운 재변환을 요청하면, 교정 내용이 반영된 마크다운 문서를 재생성하는 단계;

를 포함하며, AI 자동 변환, 사전 자동 교정, 및 인간 수동 교정의 3단계 보완 구조에 의해, 각 단계가 이전 단계의 잔여 오류를 누적적으로 보정하여 최종 99.99% 이상의 변환 정확도를 달성하는 것을 특징으로 하는 방법.

상기 인터페이스는 웹 기반 WYSIWYG HTML 에디터를 포함하며, 텍스트 직접 편집, 제목 레벨 변경, 표 셀 편집, 및 블록 순서 변경 기능을 제공하고, 사용자가 수동으로 교정해야 하는 페이지는 전체 페이지의 1~5%인 것을 특징으로 하는 방법.

### [청구항 26] (독립항 - 문서 변환·동시 번역 방법)

PDF 문서를 구조화된 문서로 변환하면서 동시에 번역을 수행하기 위한 컴퓨터 구현 방법으로서,

(a) 사용자로부터 PDF 문서의 변환 요청과 함께, 번역 수행 여부, 소스 언어, 및 타겟 언어를 포함하는 번역 옵션을 수신하는 단계;

(b) 상기 PDF 문서의 각 페이지에 대해 복잡도 태그 0(텍스트 전용) 또는 1(복합)을 부여하는 사전 분류 단계;

(c) 태그 0이 부여된 페이지에 대해, 로컬 OCR 엔진으로 텍스트를 추출하는 단계;

(d) 번역 지시를 기존의 문서 변환용 AI 프롬프트에 통합하여, 텍스트 교정, 제목 분류, 읽기 순서 결정, 및 번역을 포함하는 다중 작업을 단일 프롬프트로 구성하는 단계;

(e) 상기 태그 0 페이지에 대해, 페이지 이미지를 전송하지 않고 상기 추출된 텍스트와 번역 지시를 포함하는 단일 프롬프트를 다중모달 비전 AI에 전송하여, 텍스트 교정, 제목 분류, 및 번역을 동시에 수행하는 단계;

(f) 상기 태그 1 페이지에 대해, 페이지 이미지와 번역 지시를 포함하는 단일 다중모달 프롬프트를 AI에 전송하여, OCR, 표 구조 인식, 레이아웃 분석, 제목 분류, 텍스트 교정, 및 번역을 동시에 수행하는 단계; 및

(g) AI의 응답에서 번역된 텍스트를 포함하는 블록들을 원래의 페이지 순서로 병합하고, HTML 및 마크다운 형식의 구조화된 문서를 생성하는 단계;

를 포함하며, 문서 변환과 번역이 동일한 단일 AI 호출 내에서 수행되므로 별도의 번역 API 호출이 불필요하고, 변환 전용 대비 추가 비용이 25~35%에 불과한 것을 특징으로 하는 방법.

### 2. 핵심 발명 사상

본 발명의 핵심은 두 가지이다.

**첫째, "인간이 문서에 선을 긋는 행위의 의도를 AI가 이해하도록 하는 것"이다.**

인간이 문서에 선을 긋는 목적은 크게 두 가지이다:
1. **구조선**: "이 영역의 내용을 다 읽고 나서 다음 영역으로 이동하라"는 읽기 순서 지시
2. **지시선**: "이 주석이 저 본문 내용을 설명한다"는 참조 관계 표시

종래 기술은 이 두 가지 의도를 구분하지 못했다. 본 발명은 선의 기하학적 특성(각도)이라는 간단하면서도 본질적인 기준으로 이를 구분하고, 각각의 의도에 맞는 처리를 수행한다.

**둘째, "AI에게 필요한 것만 보내고, AI가 잘하는 일에만 집중시키는 것"이다.**

종래 기술은 모든 페이지를 일률적으로 AI에 전송하여 모든 작업을 수행시킨다. 본 발명은 페이지의 복잡도를 로컬에서 사전 판정하여, 단순 페이지(TAG=0)에는 로컬 OCR이 잘하는 텍스트 추출을 먼저 시키고 AI에는 교정만 맡기며, 복잡 페이지(TAG=1)에만 AI의 비전 능력을 집중 투입한다. 이는 인간이 업무를 위임할 때 "쉬운 건 직접 하고 어려운 것만 전문가에게 맡기는" 원리와 동일하다.

### 3. 진보성 근거
